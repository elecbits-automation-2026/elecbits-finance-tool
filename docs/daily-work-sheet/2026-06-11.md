## 5. Multi-department users with per-department dashboard tabs (PR #16)
**Commit:** `997ecd8`

A user keeps one primary `dept`, and an admin can grant additional departments
(`extra_depts`). A head with extra departments approves and sees work in all of
them; in the dashboard they switch departments with a tab bar, and a badge on
each tab shows how many of that department's items await their action.

- **Design:** the active tab re-scopes the existing dashboard to a single
  department by passing it down as `user.dept` (effectiveUser), so every view
  keeps working as if the user were single-department. Approval *routing*, by
  contrast, uses the user's full department set (`effectiveDepts()` in
  `access.ts`: primary + extras), so a request reaches a multi-department head
  regardless of which tab they have open. Single-department users see no tab
  bar and no behavioural change.
- Migration `0010_profile_extra_depts.sql`: adds `profiles.extra_depts text[]`
  (admin-managed; existing RLS covers it).
- `auth.ts` maps `extraDepts` and adds `setEmployeeExtraDepts`; the Admin
  Console gets "Additional departments" add/remove chips.
- Dashboard: department switcher bar + per-tab action badges; My Requests /
  My Approvals scoped to the active department.

**Deployment note:** migration 0010 must be applied to Supabase via the normal
deploy flow before the feature is usable in production.
